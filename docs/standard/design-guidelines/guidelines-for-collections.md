 <span data-ttu-id="17cd6-114">❌ 퍼블릭 API에서 `Hashtable` 또는 `Dictionary<TKey,TValue>`를 사용하지 마세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-114">❌ DO NOT use `Hashtable` or `Dictionary<TKey,TValue>` in public APIs.</span></span>

 <span data-ttu-id="17cd6-115">이러한 형식은 내부 구현에서 사용하도록 설계된 데이터 구조입니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-115">These types are data structures designed to be used in internal implementation.</span></span> <span data-ttu-id="17cd6-116">퍼블릭 API는 인터페이스 중 하나 또는 둘 다를 구현하는 <xref:System.Collections.IDictionary>, `IDictionary <TKey, TValue>` 또는 사용자 지정 형식을 사용해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-116">Public APIs should use <xref:System.Collections.IDictionary>, `IDictionary <TKey, TValue>`, or a custom type implementing one or both of the interfaces.</span></span>

 <span data-ttu-id="17cd6-155">속성 getter는 매우 간단한 작업이어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-155">Property getters should be very lightweight operations.</span></span> <span data-ttu-id="17cd6-156">스냅샷을 반환하려면 O(n) 작업에서 내부 컬렉션의 복사본을 만들어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-156">Returning a snapshot requires creating a copy of an internal collection in an O(n) operation.</span></span>

 <span data-ttu-id="17cd6-157">✔️ 컬렉션을 명시적으로 수정하지 않고도 변경될 수 있는 휘발성인 컬렉션을 나타내려면 스냅샷 컬렉션이나 라이브 `IEnumerable<T>`(또는 하위 형식)을 사용하세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-157">✔️ DO use either a snapshot collection or a live `IEnumerable<T>` (or its subtype) to represent collections that are volatile (i.e., that can change without explicitly modifying the collection).</span></span>

 <span data-ttu-id="17cd6-158">일반적으로 공유 리소스(예: 디렉터리의 파일)를 나타내는 모든 컬렉션은 휘발성입니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-158">In general, all collections representing a shared resource (e.g., files in a directory) are volatile.</span></span> <span data-ttu-id="17cd6-159">구현이 단순한 정방향 전용 열거자인 경우가 아니라면 이러한 컬렉션을 라이브 컬렉션으로 구현하는 것은 매우 어렵거나 불가능합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-159">Such collections are very difficult or impossible to implement as live collections unless the implementation is simply a forward-only enumerator.</span></span>

## <a name="choosing-between-arrays-and-collections"></a><span data-ttu-id="17cd6-160">배열 및 컬렉션 중에서 선택</span><span class="sxs-lookup"><span data-stu-id="17cd6-160">Choosing Between Arrays and Collections</span></span>

 <span data-ttu-id="17cd6-161">✔️ 배열보다 컬렉션을 사용하세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-161">✔️ DO prefer collections over arrays.</span></span>

 <span data-ttu-id="17cd6-162">컬렉션은 콘텐츠를 더 세부적으로 제어할 수 있으며, 시간에 따라 개선되고, 더욱 유용해집니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-162">Collections provide more control over contents, can evolve over time, and are more usable.</span></span> <span data-ttu-id="17cd6-163">또한 배열을 복제하는 비용이 너무 많이 들기 때문에 읽기 전용 시나리오에는 배열을 사용하지 않는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-163">In addition, using arrays for read-only scenarios is discouraged because the cost of cloning the array is prohibitive.</span></span> <span data-ttu-id="17cd6-164">유용성 연구에 따르면 일부 개발자는 컬렉션 기반 API를 사용할 때 더 편안하다고 합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-164">Usability studies have shown that some developers feel more comfortable using collection-based APIs.</span></span>

 <span data-ttu-id="17cd6-165">그러나 하위 수준 API를 개발하는 경우에는 읽기/쓰기 시나리오에 배열을 사용하는 것이 더 좋을 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-165">However, if you are developing low-level APIs, it might be better to use arrays for read-write scenarios.</span></span> <span data-ttu-id="17cd6-166">배열은 메모리 공간이 더 작기 때문에 작업 세트를 줄일 수 있고, 배열의 요소에 대한 액세스가 런타임에서 최적화되기 때문에 더 빨라집니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-166">Arrays have a smaller memory footprint, which helps reduce the working set, and access to elements in an array is faster because it is optimized by the runtime.</span></span>

 <span data-ttu-id="17cd6-167">✔️ 메모리 사용을 최소화하고 성능을 최대화하려면 하위 수준 API에서는 배열을 사용하는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-167">✔️ CONSIDER using arrays in low-level APIs to minimize memory consumption and maximize performance.</span></span>

 <span data-ttu-id="17cd6-168">✔️ 바이트 컬렉션 대신 바이트 배열을 사용하세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-168">✔️ DO use byte arrays instead of collections of bytes.</span></span>

 <span data-ttu-id="17cd6-169">❌ 속성 getter가 호출될 때마다 속성이 새 배열(예: 내부 배열의 복사본)을 반환해야 하는 경우에는 속성에 배열을 사용하지 마세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-169">❌ DO NOT use arrays for properties if the property would have to return a new array (e.g., a copy of an internal array) every time the property getter is called.</span></span>

## <a name="implementing-custom-collections"></a><span data-ttu-id="17cd6-170">사용자 지정 컬렉션 구현</span><span class="sxs-lookup"><span data-stu-id="17cd6-170">Implementing Custom Collections</span></span>

 <span data-ttu-id="17cd6-171">✔️ 새 컬렉션을 설계할 때는 `Collection<T>`, `ReadOnlyCollection<T>` 또는 `KeyedCollection<TKey,TItem>`에서 상속하는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-171">✔️ CONSIDER inheriting from `Collection<T>`, `ReadOnlyCollection<T>`, or `KeyedCollection<TKey,TItem>` when designing new collections.</span></span>

 <span data-ttu-id="17cd6-172">✔️ 새 컬렉션을 설계할 때 `IEnumerable<T>`을 구현하세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-172">✔️ DO implement `IEnumerable<T>` when designing new collections.</span></span> <span data-ttu-id="17cd6-173">필요한 경우 `ICollection<T>` 또는 `IList<T>`를 구현하는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-173">Consider implementing `ICollection<T>` or even `IList<T>` where it makes sense.</span></span>

 <span data-ttu-id="17cd6-174">이러한 사용자 지정 컬렉션을 구현하는 경우 `Collection<T>` 및 `ReadOnlyCollection<T>`에서 설정된 API 패턴을 최대한 비슷하게 따라야 합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-174">When implementing such custom collection, follow the API pattern established by `Collection<T>` and `ReadOnlyCollection<T>` as closely as possible.</span></span> <span data-ttu-id="17cd6-175">즉, 동일한 멤버를 명시적으로 구현하고, 이러한 두 컬렉션처럼 매개 변수의 이름을 지정하고, 해당 멤버의 이름을 지정하는 등입니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-175">That is, implement the same members explicitly, name the parameters like these two collections name them, and so on.</span></span>

 <span data-ttu-id="17cd6-176">✔️ 이러한 인터페이스를 입력으로 사용하는 API에 컬렉션을 자주 전달하는 경우 제네릭이 아닌 컬렉션 인터페이스(`IList` 및 `ICollection`)를 구현하는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-176">✔️ CONSIDER implementing nongeneric collection interfaces (`IList` and `ICollection`) if the collection will often be passed to APIs taking these interfaces as input.</span></span>

 <span data-ttu-id="17cd6-177">❌ 컬렉션의 개념과 관련이 없는 복잡한 API를 사용하여 형식에 컬렉션 인터페이스를 구현하면 안 됩니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-177">❌ AVOID implementing collection interfaces on types with complex APIs unrelated to the concept of a collection.</span></span>

 <span data-ttu-id="17cd6-178">❌ `CollectionBase`와 같은 제네릭이 아닌 기본 컬렉션에서 상속하지 마세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-178">❌ DO NOT inherit from nongeneric base collections such as `CollectionBase`.</span></span> <span data-ttu-id="17cd6-179">대신 `Collection<T>`, `ReadOnlyCollection<T>` 및 `KeyedCollection<TKey,TItem>`를 사용해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-179">Use `Collection<T>`, `ReadOnlyCollection<T>`, and `KeyedCollection<TKey,TItem>` instead.</span></span>

### <a name="naming-custom-collections"></a><span data-ttu-id="17cd6-180">사용자 지정 컬렉션 이름 지정</span><span class="sxs-lookup"><span data-stu-id="17cd6-180">Naming Custom Collections</span></span>

 <span data-ttu-id="17cd6-181">컬렉션(`IEnumerable`을 구현하는 형식)을 만드는 두 가지 주요 이유는 (1) 구조 관련 작업이 있고 종종 기존 데이터 구조와 다른 성능 특성을 갖는 새 데이터 구조(예: <xref:System.Collections.Generic.List%601>, <xref:System.Collections.Generic.LinkedList%601>, <xref:System.Collections.Generic.Stack%601>)를 만들고 (2) 특정 항목 세트(예: <xref:System.Collections.Specialized.StringCollection>)를 보유하는 특수 컬렉션을 만드는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-181">Collections (types that implement `IEnumerable`) are created mainly for two reasons: (1) to create a new data structure with structure-specific operations and often different performance characteristics than existing data structures (e.g.,  <xref:System.Collections.Generic.List%601>, <xref:System.Collections.Generic.LinkedList%601>, <xref:System.Collections.Generic.Stack%601>), and (2) to create a specialized collection for holding a specific set of items (e.g.,  <xref:System.Collections.Specialized.StringCollection>).</span></span> <span data-ttu-id="17cd6-182">데이터 구조는 애플리케이션 및 라이브러리의 내부 구현에서 주로 사용됩니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-182">Data structures are most often used in the internal implementation of applications and libraries.</span></span> <span data-ttu-id="17cd6-183">특수 컬렉션은 주로 API에서 속성 및 매개 변수 형식으로 노출됩니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-183">Specialized collections are mainly to be exposed in APIs (as property and parameter types).</span></span>

 <span data-ttu-id="17cd6-184">✔️ `IDictionary` 또는 `IDictionary<TKey,TValue>`를 구현하는 추상화 이름에 “Dictionary” 접미사를 사용하세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-184">✔️ DO use the "Dictionary" suffix in names of abstractions implementing `IDictionary` or `IDictionary<TKey,TValue>`.</span></span>

 <span data-ttu-id="17cd6-185">✔️ `IEnumerable`(또는 해당 하위 항목)을 구현하고 항목 목록을 나타내는 형식 이름에 “Collection” 접미사를 사용하세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-185">✔️ DO use the "Collection" suffix in names of types implementing `IEnumerable` (or any of its descendants) and representing a list of items.</span></span>

 <span data-ttu-id="17cd6-186">✔️ 사용자 지정 데이터 구조에 적합한 데이터 구조 이름을 사용하세요.</span><span class="sxs-lookup"><span data-stu-id="17cd6-186">✔️ DO use the appropriate data structure name for custom data structures.</span></span>

 <span data-ttu-id="17cd6-187">❌ 컬렉션 추상화 이름에 “LinkedList” 또는 “Hashtable” 같이 특정 구현을 의미하는 접미사를 사용하면 안 됩니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-187">❌ AVOID using any suffixes implying particular implementation, such as "LinkedList" or "Hashtable," in names of collection abstractions.</span></span>

 <span data-ttu-id="17cd6-188">✔️ 항목 형식의 이름을 컬렉션 이름의 접두사로 사용하는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-188">✔️ CONSIDER prefixing collection names with the name of the item type.</span></span> <span data-ttu-id="17cd6-189">예를 들어 `Address`(`IEnumerable<Address>` 구현) 형식의 항목을 저장하는 컬렉션 이름은 `AddressCollection`으로 지정해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-189">For example, a collection storing items of type `Address` (implementing `IEnumerable<Address>`) should be named `AddressCollection`.</span></span> <span data-ttu-id="17cd6-190">항목 형식이 인터페이스인 경우 항목 형식의 “I” 접두사는 생략할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-190">If the item type is an interface, the "I" prefix of the item type can be omitted.</span></span> <span data-ttu-id="17cd6-191">따라서 <xref:System.IDisposable> 항목의 컬렉션을 `DisposableCollection`이라고 할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-191">Thus, a collection of <xref:System.IDisposable> items can be called `DisposableCollection`.</span></span>

 <span data-ttu-id="17cd6-192">✔️ 해당되는 쓰기 가능 컬렉션을 추가할 수 있거나 해당 컬렉션이 프레임워크에 이미 있는 경우 읽기 전용 컬렉션의 이름에 “ReadOnly” 접두사를 사용하는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-192">✔️ CONSIDER using the "ReadOnly" prefix in names of read-only collections if a corresponding writeable collection might be added or already exists in the framework.</span></span>

 <span data-ttu-id="17cd6-193">예를 들어 문자열의 읽기 전용 컬렉션은 `ReadOnlyStringCollection`이라고 합니다.</span><span class="sxs-lookup"><span data-stu-id="17cd6-193">For example, a read-only collection of strings should be called `ReadOnlyStringCollection`.</span></span>

# <a name="pnrp-caches"></a><span data-ttu-id="3bdfa-103">PNRP 캐시</span><span class="sxs-lookup"><span data-stu-id="3bdfa-103">PNRP Caches</span></span>

<span data-ttu-id="3bdfa-104">PNRP(피어 이름 확인 프로토콜) 캐시는 한 피어에서 유지 관리되며 알고리즘 방식으로 선택된 피어 엔드포인트의 로컬 컬렉션입니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-104">Peer Name Resolution Protocol (PNRP) caches are local collections of algorithmically selected peer endpoints maintained on the peer.</span></span>  
  
## <a name="pnrp-cache-initialization"></a><span data-ttu-id="3bdfa-105">PNRP 캐시 초기화</span><span class="sxs-lookup"><span data-stu-id="3bdfa-105">PNRP Cache Initialization</span></span>  

 <span data-ttu-id="3bdfa-106">PNRP 캐시 또는 Peer Name Reord Collection을 초기화하려면 피어 노드가 시작될 때 노드에서 다음 메서드를 사용할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-106">To initialize the PNRP cache, or Peer Name Record Collection, when a peer node starts up, a node can use the following methods:</span></span>  
  
- <span data-ttu-id="3bdfa-107">노드가 종료될 때 캐시에 포함되어 있던 영구적 캐시 항목은 하드 디스크 스토리지에서 로드됩니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-107">Persistent cache entries that were present when the node was shut down are loaded from hard disk storage.</span></span>  
  
- <span data-ttu-id="3bdfa-108">애플리케이션에서 P2P 협업 인프라를 사용하는 경우 협업 정보는 해당 노드의 연락처 관리자에 있습니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-108">If an application uses the P2P collaboration infrastructure, collaboration information is available in the Contact Manager for that node.</span></span>  
  
## <a name="scaling-peer-name-resolution-with-a-multi-level-cache"></a><span data-ttu-id="3bdfa-109">다중 수준 캐시로 피어 이름 확인 확장</span><span class="sxs-lookup"><span data-stu-id="3bdfa-109">Scaling Peer Name Resolution with a Multi-Level Cache</span></span>  

 <span data-ttu-id="3bdfa-110">피어 노드에서는 PNRP 캐시의 크기를 작게 유지하기 위해 다중 수준 캐시를 사용합니다. 이러한 다중 수준 캐시의 각 수준에는 최대 개수의 항목이 들어 있습니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-110">To keep the sizes of the PNRP caches small, peer nodes use a multi-level cache, in which each level contains a maximum number of entries.</span></span> <span data-ttu-id="3bdfa-111">캐시의 각 수준은 PNRP ID 번호 공간(2<sup>256</sup>)의 1/10에 해당하는 부분을 나타냅니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-111">Each level in the cache represents a one tenth smaller portion of the PNRP ID number space (2<sup>256</sup>).</span></span> <span data-ttu-id="3bdfa-112">캐시의 최하위 수준에는 로컬로 등록된 PNRP ID와 수치상 해당 PNRP ID에 가까운 다른 PNRP ID가 들어 있습니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-112">The lowest level in the cache contains a locally registered PNRP ID and other PNRP IDs that are numerically close to it.</span></span> <span data-ttu-id="3bdfa-113">캐시의 특정 수준이 최대 항목 수인 20개의 항목으로 채워지면 새로운 하위 수준이 생성됩니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-113">As a level of the cache is filled with a maximum of 20 entries, a new lower level is created.</span></span> <span data-ttu-id="3bdfa-114">캐시에 생성될 수 있는 수준의 최대 수는 log10(클라우드의 총 PNRP ID 수)의 차수에 해당합니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-114">The maximum number of levels in the cache is on the order of log10(Total number of PNRP IDs in the cloud).</span></span> <span data-ttu-id="3bdfa-115">예를 들어 PNRP ID가 1억개 포함된 글로벌 클라우드의 경우 캐시에 수준이 8개(=log10(100,000,000))까지만 생성될 수 있으며 이름 확인 작업을 수행하는 동안 PNRP ID를 확인하는 홉의 개수도 이와 비슷하게 됩니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-115">For example, for a global cloud with 100 million PNRP IDs, there are no more than 8 (=log10(100,000,000)) levels in the cache and a similar number of hops to resolve a PNRP ID during name resolution.</span></span> <span data-ttu-id="3bdfa-116">이러한 메커니즘을 활용하면 해당 CPA가 있는 피어를 찾을 때까지 PNRP 요청 메시지를 다음으로 가까운 피어에 전달함으로써 임의 PNRP ID를 확인할 수 있는 분산된 해시 테이블을 만들 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-116">This mechanism allows for a distributed hash table for which an arbitrary PNRP ID can be resolved by forwarding PNRP Request messages to the next-closest peer until the peer with the corresponding CPA is found.</span></span>  
  
 <span data-ttu-id="3bdfa-117">항상 이름 확인이 가능하도록 하기 위해 노드에서는 캐시의 최하위 수준에 항목을 추가할 때마다 항목의 복사본을 캐시의 마지막 수준 내에 있는 모든 노드로 전파합니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-117">To ensure that resolution can complete, each time a node adds an entry to the lowest level of its cache, it floods a copy of the entry to all the nodes within the last level of the cache.</span></span>  
  
 <span data-ttu-id="3bdfa-118">캐시 항목은 시간이 지남에 따라 새로 고쳐지고</span><span class="sxs-lookup"><span data-stu-id="3bdfa-118">The cache entries are refreshed over time.</span></span> <span data-ttu-id="3bdfa-119">오래된 항목은 캐시에서 제거됩니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-119">Cache entries that are stale are removed from the cache.</span></span> <span data-ttu-id="3bdfa-120">따라서 주소 레코드와 DNS 프로토콜에서 주소에 연결된 노드가 네트워크에서 작동 중인지 여부가 보장되지 않는 DNS와는 달리 PNRP ID의 분산된 해시 테이블은 현재 엔드포인트를 기준으로 합니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-120">The result is that the distributed hash table of PNRP IDs is based on active endpoints, unlike DNS in which address records and the DNS protocol provide no guarantee that the node associated with the address is actively on the network.</span></span>  
  
## <a name="other-pnrp-caches"></a><span data-ttu-id="3bdfa-121">다른 PNRP 캐시</span><span class="sxs-lookup"><span data-stu-id="3bdfa-121">Other PNRP Caches</span></span>

 <span data-ttu-id="3bdfa-128">피어가 클라우드에 조인할 때, 연결된 호스트 피어에서 자동으로 로컬 캐시 데이터를 받지 않습니다. 애플리케이션, 현재 상태 및 개체 데이터의 업데이트를 받으려면 호스트 피어에 가입해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-128">When a peer joins a cloud, they do not automatically receive local cache data from the host peer to which they attach; they have to subscribe to the host peer to receive updates in application, presence, and object data.</span></span> <span data-ttu-id="3bdfa-129">모든 그룹 멤버에게 일관되게 동일한 보기가 표시되도록 초기 동기화 후에 피어에서 복제된 저장소를 정기적으로 다시 동기화합니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-129">After the initial synchronization, peers periodically resynchronize their replicated stores to ensure that all group members consistently have the same view.</span></span>  <span data-ttu-id="3bdfa-130">협업 세션 또는 협업 세션의 애플리케이션에서도 동일한 기능을 수행할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="3bdfa-130">The collaboration session or applications within the collaboration session may also perform the same function.</span></span>
